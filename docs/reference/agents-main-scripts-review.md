> **agents-main — scripts review.** Not a markdown collection: it ships a genuinely production-grade Python build/transpile/eval toolchain (`tools/` adapter framework + `plugin-eval/` package) sitting above a thin layer of demo-grade in-skill asset scripts — judge the toolchain, footnote the demos. Analyzed 2026-06-20 · 9.4M · source: local vendored. Scope: scripts. Method: local read-only sampling (~12 of 59 scripts read end-to-end + targeted scans across tools/, plugins/plugin-eval/, and scattered in-plugin scripts) → scripts-specialist synthesis.

## 1. Spine — the headline judgment

The intake hypothesis ("largely a subagent/markdown collection, real executable code may be thin") is **wrong for the scripts domain**. agents-main is a 84-plugin multi-harness marketplace whose *delivery mechanism* is real software: a single-source-of-truth `plugins/` tree is transpiled to five harnesses (Codex/Cursor/OpenCode/Gemini/Copilot) by a clean adapter framework under `tools/`, gated by three mechanical CI checks (`validate` / `garden` / `test`), and a separate installable `plugin-eval` Python package scores skill quality with a static→LLM-judge→Monte-Carlo pipeline. 266 test functions back it. This is the **best executable code reviewed across the harness fleet so far** — and it earns that by being ordinary, disciplined Python, not by being clever.

The spine lesson is **"single-source the contract, multi-source the delivery, and make the build itself a guarded program"** — the same lesson Trellis taught, but here the build tooling is the *primary* artifact rather than a side-effect. Crucially, the repo's safety posture is encoded **in code, not rhetoric**: `generate.py` refuses to wipe paths outside the repo/tmp (`_validate_output_root`, generate.py:66-114), refuses `--clean --plugin` because it would silently delete other plugins' artifacts (generate.py:234-240), and `base.py`'s `write()` rejects any path that escapes `output_root` via `..` or absolute (base.py:428-434). Errors are **aggregated and surfaced with tracebacks, never swallowed** (generate.py:283-287). That is exactly the failure-posture nxtlvl's LOCKED positions ask for.

The honest caveat: this is **two tiers in one repo**. Tier 1 (`tools/`, `plugin-eval/src/`) is excellent. Tier 2 — the per-skill helper scripts shipped *inside* plugins as teaching assets (`validate-chart.sh`, `optimize-prompt.py`, the `rest-api-template.py`) — is demo-grade: `set -e` bash with no `mktemp`/trap discipline, an `optimize-prompt.py` that takes an undefined `llm_client` and is never wired to anything. They are documentation-by-example, not load-bearing code, and should be read as such.

## 2. What's there & how it works (inventory + end-to-end, cited)

**Counts:** 56 `*.py`, 3 `*.sh`, 0 `*.js`, 686 `*.md`. The Python splits into three populations:

- **`tools/` — the transpiler/build framework (load-bearing).**
  - `adapters/base.py` — shared spine: a dependency-free tolerant frontmatter parser (`parse_frontmatter`, base.py:36-129, handles inline+block lists, block scalars, quoted-comma items), source dataclasses (`AgentSource`/`SkillSource`/`CommandSource`/`PluginSource` with typed `@property` accessors, base.py:220-312), the `HarnessAdapter` ABC with safe `write`/`write_bytes`/`mirror_file` (base.py:388-451), and `load_plugin` which **rejects plugin names containing `__`** because that's the cross-harness namespace separator (base.py:314-331) — a real invariant defended in code.
  - `adapters/{codex,cursor,opencode,gemini,copilot}.py` + `capabilities.py` — per-harness emitters behind one ABC; `generate.py` lazy-imports them (generate.py:41-63) so the CLI stays fast for a single target.
  - `generate.py` — the `make generate` CLI. End-to-end: parse args → containment-guard if destructive → optional `clean_output` → `load_plugin` per target → `adapter.emit_plugin` (exceptions aggregated per-plugin, generate.py:281-287) → `adapter.emit_global` for manifests → optional `prune_orphans` (only with `--all`, "the only case where we have a complete view," generate.py:313-324) → tallied summary → exit 1 on any error, exit 1 on warnings only under `--strict` (generate.py:332-336).
  - `validate_generated.py` + `doc_gardener.py` — the other two gates. Both use a `Finding(severity, path, message, fix/remediation)` dataclass so **every machine finding carries a concrete fix string** (validate_generated.py:35-44, doc_gardener.py:49-64) — invariant #4 from ARCHITECTURE.md enforced uniformly. Gardener checks stale-mtime artifacts, oversize context files, dead doc→plugin links, >8KB skills without `references/`, and marketplace/dir mismatches (doc_gardener.py:1-44).
  - `check_agent_name_collisions.py`, `install_opencode.py`, `install_copilot.py` — supporting CLIs.
- **`plugins/plugin-eval/` — an installable evaluation package (load-bearing, separate product).** `src/plugin_eval/` is a proper `uv`/Typer package: `cli.py` (Typer app, score/certify/init/compare commands, cli.py:92-176), `engine.py`, `models.py`, `reporter.py` (json/markdown/html), `corpus.py`, `elo.py`, `stats.py`, and `layers/{static,judge,monte_carlo,harness_portability}.py`. The judge layer (`layers/judge.py`) runs 4 async assessments concurrently via `asyncio.gather` under a `Semaphore` (judge.py:130-168), uses **anchored 0.0-1.0 rubrics** (judge.py:18-32), resolves model tiers to current IDs (judge.py:38-47), and **fails loud** if `claude-agent-sdk` is missing (`raise RuntimeError(...)`, judge.py:66-69) rather than faking a score. `scripts/eval_all.py` batch-runs it via the library (not subprocess) for CI.
- **In-plugin helper scripts (demo-grade).** `optimize-prompt.py` (depends on an unprovided `llm_client`, optimize-prompt.py:24), `validate-chart.sh` (`set -e`, emoji output, helm wrapper — works but ad-hoc), `rest-api-template.py` (an asset, not a tool). `yt-design-extractor/` is the one substantive standalone tool (its own uv project with a Makefile).

**Tests:** 266 test functions across `plugins/plugin-eval/tests/` (14 files) and `tools/tests/` (8 files incl. `test_round_trip.py`, `test_real_world.py`, `test_cli_smoke.py`). The 2 bash test harnesses (`protect-mcp/test/run-tests.sh`) are notably **better-engineered than the in-skill demos**: `set -uo pipefail`, `mktemp -d` + `trap 'rm -rf' EXIT`, a `need()` preflight that exits **77 (automake "skipped")** on missing tools, TTY-gated color, and explicit per-test exit-code assertions incl. tamper-detection expecting exit 1 (run-tests.sh:1-178).

**Hygiene scan:** no secrets in executable code (the lone `api_key`/`secret` grep hit is `cursor_rules/000-project-conventions.mdc`, a prose rules file). Shebangs present on all CLIs; `from __future__ import annotations` throughout; `sys.path.insert` bootstrap is the one mild smell but is standard for run-from-root tooling.

**Strengths.** Safety-in-code not safety-in-prose (containment guards, path-escape rejection, footgun refusals). Uniform observation contract — every gate finding ships its own fix. Honest failure posture end to end: loud raises, meaningful exit codes (incl. 77-skip and tamper→1), no fabricated clean exits — the exact anti-pattern that disqualified ruflo-adr's `verify.mjs` is *absent* here. Real test depth (266 functions, round-trip + real-CLI smoke). Dependency-light core (hand-rolled tolerant YAML parser).

**Weaknesses & risks.** (1) **Two-tier quality** — the in-skill demo scripts (`optimize-prompt.py` unwired to any client; `validate-chart.sh` ad-hoc `set -e` bash) are weaker than the toolchain and a naive reader could mistake them for the repo's standard; they're best understood as documentation assets. (2) `sys.path.insert(0, ...)` bootstrap in every `tools/` CLI is a mild packaging smell (run-from-root convention rather than an installed console-script). (3) The `parse_frontmatter` tolerant parser is genuinely good but is bespoke — correctness rests on its own test suite rather than a spec'd YAML subset, so exotic frontmatter could parse surprisingly. None are architectural; all are local.

**Headline verdict.** For the scripts domain, agents-main is the strongest executable code in the reviewed fleet: a disciplined, well-tested Python transpiler/build framework plus a separate evaluation package, both exhibiting precisely the properties nxtlvl's LOCKED positions prize — safety encoded in code, uniform fix-carrying observation, honest exit codes, loud failure, no lying clean exits, files-over-DB. There is no fatal D3/D4 flaw to cap it; the dominant dimensions are its strongest. The only real care a reader must take is to not generalize from the thin tier-2 in-skill demo scripts to the load-bearing toolchain — judge it by `tools/` and `plugin-eval/src/`, which set the bar. Mine the patterns (containment-guarded destructive ops, `Finding`-with-fix observation contract, ABC-per-harness lazy dispatch, 77-skip bash harness); the in-plugin asset scripts are not worth mining.
