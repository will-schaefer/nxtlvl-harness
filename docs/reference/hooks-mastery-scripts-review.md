## 1. Spine — the headline judgment

This is a **demo repo whose script layer quietly over-delivers on craft**. The README sells it as "quickly master CC hooks," and the surface reads like hello-world filler — yet the actual executable code is consistent, defensively written, and wires *every* hook event (`settings.json:24-166` covers all 13 event types, each to a single-job handler). The dominant dimensions both score well: observation (D3) ranges from adequate (bare exit-0 demo loggers) to **exemplary** (`validate_new_file.py:51-56` `NO_FILE_ERROR` tells the caller the exact failure *and* the next action), and the error/exit-code contract (D4) is the repo's strongest property — a **disciplined, uniform fail-open posture** for observer hooks (`pre_tool_use.py:131-136`, `user_prompt_submit.py:183-188`: `JSONDecodeError → exit 0`, `Exception → exit 0`) with **honest, role-appropriate exit-2 blocking** reserved for the one place a hook *should* intervene (the security gate). Two leaf scripts (`tts_queue.py`, the validators) are outright production-grade and are the real harvest. The one thing that caps it: **all 13 logging hooks read-the-whole-JSON-array → append → rewrite-the-whole-file** (`pre_tool_use.py:113-127`, replicated verbatim ×13) — non-atomic, race-prone under concurrent/subagent hooks, and O(n) growth. That's a genuine side-effect-safety (D5) flaw, not demo filler, and it's the load-bearing weakness of an otherwise clean corpus.

## 2. What's there & how it works

~48 scripts total; the load-bearing executable layer is ~25 Python files under `.claude/hooks/`. All share one entrypoint convention: PEP-723 inline-dependency `uv run --script` single-file scripts (`#!/usr/bin/env -S uv run --script` — uniform across all 25, verified by grep). Invocation is `uv run $CLAUDE_PROJECT_DIR/.claude/hooks/<handler>.py [flags]` (`settings.json`).

| Layer | Files | Role | End-to-end |
|---|---|---|---|
| Event hooks | 13 (`pre_tool_use`, `post_tool_use`, `stop`, `session_start`, `user_prompt_submit`, …) | One handler per CC event, all wired in `settings.json:24-166` | Read stdin JSON → extract fields with `.get(...,default)` → append to `logs/<event>.json` → optionally act (block / emit context / TTS) → `sys.exit(0)` |
| Validators | 4 (`ruff_validator`, `ty_validator`, `validate_new_file`, `validate_file_contains`) | PostToolUse code-quality gates | Run external tool (`uvx ruff`, `git status`) → emit `{"decision":"block","reason":...}` or `{}` |
| LLM utils | 4 (`anth`, `oai`, `ollama`, `task_summarizer`) | Library + CLI for name/completion generation | `prompt_llm()` returns text or `None`; CLI dispatch on `sys.argv[1]` |
| TTS utils | 4 (`tts_queue`, `elevenlabs_tts`, `openai_tts`, `pyttsx3_tts`) | Cross-process audio queue + provider backends | `tts_queue` exposes fcntl-locked acquire/release; provider scripts speak one message |

**Control flow that matters (claims-vs-wiring):**
- **Security gate** (`pre_tool_use.py`): regex-detects dangerous `rm -rf` (`:11-52`) and `.env` access (`:54-82`), `exit(2)` with stderr to block (`:94-105`). README claim "Blocks dangerous commands (`rm -rf`, `.env` access)" (`README.md:110,172`) — **matches wiring exactly.** (Detection is heuristic/regex and bypassable — `$HOME`-expansion, `find -delete`, etc. — but that's the demo's stated scope, not a dishonest claim.)
- **PostToolUse decision contract** (`ruff_validator.py:76-99`): `exit 0`+`{}` on pass, `{"decision":"block","reason":...}` on lint failure, **and explicitly `{}` (allow) when `uvx ruff` is `FileNotFoundError`** (`:96-98`) — correct fail-open for a missing optional tool. Matches the documented exit-code table (`README.md:294-302`).
- **Two-tier LLM/TTS fallback** (`stop.py:65-129`, `user_prompt_submit.py:75-109`): OpenAI→Anthropic→Ollama→static-message, each guarded by API-key presence + `subprocess` timeout + `except: pass`. Degrades silently to a hardcoded message; never blocks the agent.

**Strengths (evidence)**
- **Fail-open discipline is uniform and intentional**, not accidental — the same `except JSONDecodeError: exit(0)` / `except Exception: exit(0)` epilogue appears in every observer hook (`pre_tool_use.py:131-136`, `user_prompt_submit.py:183-188`, `session_start.py:204-209`, `stop.py:223-228`). For hooks, this is the *correct* posture: an observer must never crash the agent.
- **Honest blocking exit codes**: `exit(2)` appears in exactly the two places intervention is warranted (security gate + optional prompt validation), each with stderr the model consumes (`pre_tool_use.py:96,105`; `user_prompt_submit.py:173-174`). The README exit-code table (`README.md:294-302`) is faithfully implemented.
- **Two production-grade leaf scripts** worth mining: `tts_queue.py` (fcntl cross-process lock, `os.kill(pid,0)` liveness check for orphan reclaim, exponential backoff — `:68-229`) and the validator family (parseable decision contract + action-oriented error strings).
- **Injection-safe by construction**: every subprocess is an argv list with interpolated values as discrete args (`ruff_validator.py:63`, `session_start.py:53-57,88-93`) — no shell, so no command-injection surface even though tool inputs flow in.

**Weaknesses & risks**
- **Systemic non-atomic log writes (caps D5 at 2):** the read-whole-array → append → rewrite-whole-file pattern is duplicated across all 13 event hooks (`pre_tool_use.py:113-127`). Under parallel subagent hooks (this repo *ships* SubagentStart/Stop hooks) two handlers can interleave read/write and lose entries or write truncated/corrupt JSON; there is no `os.replace`/tempfile/`flock` guard on these writes (grep: only `tts_queue.py` uses flock). It also grows unbounded and re-serializes the full history every event. This is real production-code debt, not teaching simplification.
- **Untyped input contract (D1):** fields are plucked positionally with defaults; a malformed-but-valid-JSON stdin silently yields empty defaults rather than a diagnosable error. Acceptable for fail-open observers, but it's the dimension a production harness would harden.
- **Heuristic security gate is bypassable** (`pre_tool_use.py:11-52`): regex-based `rm -rf` detection won't catch `find -delete`, env-var-obscured paths, or non-`rm` destructive tools. *Not a claim-vs-wiring gap* — the README scopes this as a demonstration — but do not mistake it for a real safety boundary.
- No fatal claim-vs-wiring dishonesty found: documented behaviors (block dangerous cmds, exit-code semantics, decision JSON) all match shipped code.

**Headline verdict.** On its own terms this is **better-than-demo script craft**: the hygiene (D6=5) is spotless, the exit-code/error posture (D4=4) is exactly what a hook layer should do — fail-open for observers, honest exit-2 only where intervention is legitimate, no lying clean exits over real work — and the observation quality (D3=4) peaks at genuinely exemplary in the validators. The reusable signal (as opposed to hello-world filler) is concentrated in four places worth lifting: the uniform fail-open epilogue, the validator decision-contract + action-oriented error strings, the argv-list injection-safe subprocess idiom, and `tts_queue.py`'s fcntl-lock-with-liveness-reclaim. What caps the corpus is a single systemic side-effect flaw (D5=2): a non-atomic, race-prone, unbounded full-rewrite logging pattern copy-pasted into all 13 event hooks — the one place the demo's "just log it" convenience hardened into a real defect that a production harness must replace with append-only or atomic writes. Net: high-craft demonstrative code with one load-bearing weakness; mine the four patterns, reject the logging idiom.
