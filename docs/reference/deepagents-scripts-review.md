> **deepagents — scripts review.** LangChain's "batteries-included" Python agent harness built on LangGraph — a code-runtime where the harness IS the library, and the load-bearing tool-I/O code is some of the most disciplined error-and-observation contract work in the reference set. Analyzed 2026-06-20 · 45M · source: local vendored. Scope: scripts. Method: local read-only sampling fan-out (~10 modules read in full + targeted greps over ~675 files) → scripts-specialist synthesis.

## 1. Spine — the headline judgment

deepagents is not a `.claude/`-style config harness; it is a **published Python library monorepo** (`libs/{deepagents,cli,code,acp,talon,evals,partners/*}`) where capability lives entirely in code. The headline package is `libs/deepagents/deepagents/` (~20.3K LOC), and its spine is a single factory — `create_deep_agent` (`graph.py:236`) — that assembles a LangGraph agent from an ordered middleware stack (filesystem, subagents, summarization, skills, memory, permissions, prompt-caching).

The spine judgment: **the contract discipline is the product.** Every tool-facing operation returns a *typed result object with an explicit `error` field* rather than raising into the agent loop or returning a bare string — `ReadResult`/`WriteResult`/`EditResult`/`LsResult`/`GrepResult`/`GlobResult`/`ExecuteResponse` (`backends/protocol.py:184–324,780`), each one a `@dataclass` or `TypedDict` with documented success/failure shapes. The filesystem backend (`backends/filesystem.py`, 1102 lines) reads almost like a hardening spec: `O_NOFOLLOW` on every open, ELOOP symlink-loop re-raising across Python-version behavior changes (`:1081`), `root_dir`-leak scrubbing in agent-visible errors (`_safe_detail`/`_display_path`, `:790,237`), partial-result flagging on every mid-walk abort (`:874`), and ripgrep exit-2 explicitly routed to the Python fallback instead of being parsed as "zero matches" (`:677`) — the exact silent-failure trap called out in a comment. This is claims-match-wiring code: the README's "trust the LLM, enforce at the tool/sandbox boundary" security model is actually implemented at the boundary, not asserted in prose.

**Strengths (evidence):**
- Typed result-object contract is uniform and total — there is no code path in the sampled backends that returns an untyped blob or raises into the agent loop (`protocol.py`, `filesystem.py`).
- Silent-failure aversion is explicit doctrine in the code, not incidental: ripgrep exit-2 (`:677`), partial-walk flags (`:874`), one-time INFO on missing `rg` (`:61`) — each with a comment naming the failure it prevents.
- Security model is wired, not asserted: `O_NOFOLLOW`, virtual-mode traversal blocking (`:197`), root-path scrubbing from agent-visible errors, protected-scaffolding `ValueError` guard.
- Honest exit/observation surface on the destructive `execute` path: real exit codes, `[stderr]` attribution, truncation flag, `stdin=DEVNULL`.

**Weaknesses & risks (claim-vs-wiring):**
- **Non-atomic edit/write** (`filesystem.py:558–563`, `:497`) — the genuine D5 gap. A crash between truncate and write loses data; a temp-file-plus-rename would make it crash-safe. Bounded (FilesystemBackend is documented dev-only, not the production default), so it caps D5 at 4 rather than the whole review.
- **Deprecation-default model** (`graph.py:146` → `claude-sonnet-4-6`) is a soft footgun, but it warns loudly and is on a 1.0.0 removal path — handled honestly.
- No claim-vs-wiring deception found: docstrings, README security posture, and the shipped code agree. The "batteries-included / production-ready" README claim is the one place to caveat — `FilesystemBackend`/`LocalShellBackend` are explicitly NOT the production answer (their own docstrings say use `StateBackend`/`SandboxBackend`), so "production-ready" applies to the *architecture*, not every default backend.

**Headline verdict:** This is reference-grade executable agent-harness code — the strongest tool-I/O contract discipline in the set so far. The dominant dimensions (D3 observation, D4 error/exit) are both 5s with no fatal flaw: results are typed and total, errors carry a next-step, exit codes are honest, and the code actively refuses the silent-failure and lying-clean-exit traps (ripgrep exit-2 → fallback, partial walks flagged, degraded-agent → `ValueError`). What it is NOT is a config-style harness to lift wholesale; the harvest is *patterns* — the typed `error`-field result object per operation, the partial-result-must-be-flagged rule, the protected-scaffolding hard-fail, and the `[stderr]`-attributed honest `execute` envelope. The only substantive risk is non-atomic file writes (D5), which is real but bounded to a backend its own docstring tells you not to run in production. Overall: ~4.7/5, capped just short of perfect only by the missing write atomicity.
