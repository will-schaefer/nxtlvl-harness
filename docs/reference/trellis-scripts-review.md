> **Trellis — scripts review.** A genuinely senior-grade Python CLI engine — single-sourced, incident-hardened, honest about its own degraded modes — whose one consistent craft gap is non-atomic file writes (bare `write_text`, never `os.replace`) in a tool whose entire job is mutating shared state files. Analyzed 2026-06-20 · 130M · source: local vendored. Scope: scripts. Method: local read-only sampling fan-out (~16 of ~71 Python + spot-checks of the TS/JS surface) → scripts-specialist synthesis.

## 1. Spine — the headline judgment

Trellis's script layer is a **single-sourced Python workflow engine** (`.trellis/scripts/`, ~22 modules) that scaffolds an AI agent's spec→task→workflow loop, plus a **per-turn breadcrumb hook** (`inject-workflow-state.py`) shared across 8+ AI platforms, plus a **TypeScript installer** (`packages/cli/src/`, 83 non-template files) that writes the Python engine into a target repo. The Python is the load-bearing layer and it is **high-craft**: typed signatures (`from __future__ import annotations`, frozen dataclasses, `dict | None` returns), one-job-per-module decomposition, argv-list subprocess (injection-safe), honest 0/1/2 exit codes, and — the standout — **incident-driven defensive design with the incident written into the docstring** (`safe_commit.py:6-30` reconstructs a real 548-file/83k-line bad-commit and forbids the `git add -f` retry that caused it). The prior Mode-B claims hold under code inspection: single-source is **byte-verified** (live `.trellis/` and `.claude/hooks/` are `diff`-identical to the `packages/cli/.../templates/` originals), and the **non-atomic-write counterexample is real and pervasive** — every one of ~10 write sites uses bare `write_text`, zero use `os.replace`. The spine lesson: *this is what spec/task/workflow CLI craft looks like when it's dogfooded* — and the single durability gap is the one a reviewer should not let the quality halo hide.

## 2. What's there & how it works

Inventory (Python engine + hooks; the TS CLI is the installer, scored only at the margins):

| Layer | Files | Role |

| Session-state core | `common/active_task.py` (627 L) | resolves the per-session active-task pointer under `.trellis/.runtime/sessions/`; multi-platform session-key detection (stdin JSON → env vars → cursor-shell ticket) |
| Git safety | `common/safe_commit.py`, `common/git.py` | narrow-scope `git add`, never `-f`; argv-list `run_git` |
| Helpers | `common/io.py`, `paths.py`, `log.py`, `task_store.py`, `task_context.py`, `task_utils.py`, `tasks.py`, `config.py`, `trellis_config.py`, `developer.py`, … | single-source JSON I/O, path constants, ANSI color, task-CRUD split modules |
| Per-turn hook | `inject-workflow-state.py` (364 L) + `session-start.py`, `inject-subagent-context.py`, `statusline.py`, `linear_sync.py` | UserPromptSubmit breadcrumb; emits `<workflow-state>` from `workflow.md` tag blocks |
| TS installer | `packages/cli/src/**` (83 files: per-platform `configurators/`, `migrations/`, `utils/`) | scaffolds the Python engine + per-platform hook wiring into a repo |

End-to-end (the active-task flow, cited): `task.py start <dir>` (`task.py:70`) resolves repo root, calls `resolve_context_key()` (`active_task.py:380`) which tries `TRELLIS_CONTEXT_ID` env → stdin session/conversation/transcript ids → platform env vars → a short-lived Cursor shell ticket; if a key is found it writes `current_task` into `.runtime/sessions/<key>.json` (`set_active_task`, `active_task.py:548`). On every subsequent prompt, `inject-workflow-state.py:303` reads stdin, walks up for `.trellis/` (`find_trellis_root:73`), re-resolves the active task, parses `workflow.md` `[workflow-state:STATUS]` tag blocks (`load_breadcrumbs:174`), and prints a `<workflow-state>` block via `hookSpecificOutput.additionalContext` — platform-aware (`BeforeAgent` for Gemini 0.40.x, else `UserPromptSubmit`, `:348`). The genuinely impressive bit is the **honest degraded mode**: when no session identity is resolvable, `task.py:95-119` prints a loud "degraded mode" notice, still flips `planning→in_progress`, and returns 0 — and `resolve_active_task` (`active_task.py:468`) refuses to guess across windows (returns a `session-fallback` only when **exactly one** session file exists, `:497-519`).

**Strengths (evidence).**
- **Incident-as-docstring** defensive design: `safe_commit.py:6-30` narrates a real 548-file bad-commit, forbids the `git add -f .trellis/` retry, and even bakes the negative example into the user-facing warning (`:309`) "so any AI rereading the log doesn't reinvent the bug." This is the single best piece of script craft in the harness set.
- **Honest failure posture** (D3+D4, the dominant pair): degraded mode is announced and recoverable, never a silent clean exit; the multi-window isolation contract *refuses to guess* (`active_task.py:497-519`) rather than risk cross-session pointer contamination.
- **Verified single-source** (D7): `diff` proves `.claude/hooks/inject-workflow-state.py`, `.codex/hooks/...`, and `.trellis/scripts/{task,common/io}.py` are byte-identical to their `packages/cli` template originals — the "single-source the contract, multi-source the delivery" claim is real, not aspirational.
- **Clean stdout/stderr contract** (D2): parseable data to stdout, all human chatter to stderr — `add_session.py` writes *every* progress line to stderr (`:452-492`), leaving stdout uncontaminated.

**Weaknesses & risks.**
- **Non-atomic writes (capping flaw, D5).** Verified pervasive, not isolated: 10 write sites, bare `write_text`, zero `os.replace`/temp-then-rename. The most-exposed file is the per-session `.runtime/sessions/<key>.json` pointer (`active_task.py:429`) written on every `task.py start` — a crash or full disk mid-write corrupts the active-task pointer the per-turn hook depends on. The mitigating grace is that reads fail-soft (`io.py:21`, `active_task.py:421` catch `JSONDecodeError`→`None`), so a corrupt file degrades to "no active task" rather than a crash — but that silently *loses* the pointer. For a tool whose entire purpose is durable shared-state mutation, the atomic-write primitive should have been the default, and its absence is the one thing capping an otherwise 5-grade layer.
- **Duplicated I/O primitive (minor).** `active_task.py:418-435` carries its own private `_read_json`/`_write_json` instead of importing the "single source of truth" `io.py` pair — a small dent in the otherwise-clean D7 single-sourcing, and it means the atomicity fix would need to land in two places.
- **Claim-vs-wiring: no gaps found.** Unlike the ruflo/SuperClaude reviews, docs here match wiring — the README's spec/task/workflow claims are backed by runnable code, hooks are registered, and the degraded-mode docstrings describe what the code actually does.

**Headline verdict.** This is the highest-quality script layer reviewed to date — senior-engineer Python with typed contracts, one-job modules, injection-safe subprocess, genuinely honest exit codes and observation messages, and incident-hardened git safety, all single-sourced and byte-verified into multi-platform delivery. The dominant D3/D4 pair is a clean 5/5: the engine always tells its caller what happened and what to do next, and it never fakes a clean exit. What caps it is **D5 side-effect durability** — the uniform use of bare `write_text` with no `os.replace` anywhere in ~10 write sites means the one thing this tool exists to do (mutate shared workflow-state files safely) is done non-atomically, so a mid-write crash can corrupt the very session pointer the per-turn hook reads. The quality halo makes that gap easy to miss, which is exactly why it's the load-bearing finding: adopt the craft (incident-as-docstring, honest degraded modes, single-source + byte-verify, narrow-scope git staging), but treat its file-write primitive as the counterexample, not the model.
